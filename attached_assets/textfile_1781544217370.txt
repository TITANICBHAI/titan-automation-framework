### 4. TECHNICAL MODULE DESIGN & MECHANICS
#### 4.1 Accessibility Automation Layer (MacroAccessibilityService.kt)
 * **Under-the-Hood Mechanics**: Extend Android's native AccessibilityService. By configuring the service with android:canPerformGestures="true", the framework can synthesize touch inputs on behalf of the user.
 * **Input Simulation**: Inject complex inputs via dispatchGesture() using GestureDescription.Builder.
 * **Bezier Curve Generation**: Linear movements are highly susceptible to anti-bot detection algorithms monitoring pointer velocity vectors. All swipes must be translated into randomized Cubic Bezier curve paths, injecting progressive acceleration/deceleration trends using the equation:
 * **Gaussian Micro-Offsets**: Apply Gaussian noise (with a configurable standard deviation \sigma \in [2, 8] pixels) to target touch coordinates, simulating natural human precision variance.
 * **Bypassing App Restrictions**: Standard window overlays are blocked by target applications implementing filterTouchesWhenObscured to prevent tap-jacking. Because accessibility gestures are dispatched directly from the system's internal framework rather than local window elements, they bypass these security filters.
#### 4.2 High-Efficiency Screen Capture Pipeline (ScreenCaptureService.kt)
 * **Under-the-Hood Mechanics**: To avoid the memory pressure of continuously generating large Bitmap objects in Java memory, the framework uses a zero-copy screen capture mechanism.
 * **Compositor Access**: On Android 11 (API 30) or higher, the framework can acquire screen contents directly from the hardware compositor via AccessibilityService.takeScreenshot() by enabling the android:canTakeScreenshot="true" XML parameter.
 * **Fallback Capture**: On Android 10, the pipeline falls back to a background MediaProjection foreground service, streaming frames into an ImageReader configured to utilize direct HardwareBuffer allocations.
 * **JNI Bridge Mapping**: Pointers to the raw underlying AHardwareBuffer are mapped to native OpenCV structures via a JNI layer to prevent JVM memory allocations and garbage collection pauses:

#### 4.3 Multimodal Vision Engine (VisionEngine.kt)
 * **Traditional CV**: Initialize native OpenCV bindings locally. Implement template matching (Imgproc.matchTemplate) with support for normalized cross-correlation algorithms (TM_CCOEFF_NORMED) to compute match accuracy values. Support multi-scale template resizing to handle varying screen layouts.
 * **OCR Subsystem**: Integrate **Google ML Kit Text Recognition v2** for on-device OCR. Restrict OCR operations to targeted regions of interest (ROIs) on the screen to prevent frame queue bottlenecks and conserve CPU cycles.
 * **Lightweight ML Inference**: Integrate TensorFlow Lite to run classification models. Ensure the models are quantized to 8-bit integers (INT8) to reduce memory usage and allow them to run efficiently on lower-end mobile processors.
#### 4.4 Staged Reinforcement Learning Subsystem (RLEngine.kt)
 * **Under-the-Hood Mechanics**: Standard automation rules break when layout items shift dynamically. To make the framework adaptive, implement an on-device Q-learning state coordinator modeled after a dual-agent structure:

   * *The Interactor*: Converts those targets into precise touch and swipe gestures.
 * **Action Masking**: Apply action masks to target interfaces. The system only evaluates actions that land on valid, interactive regions of the screen, ignoring flat background elements to improve decision speed.

#### 4.5 Execution Engine and DSL Compiler (MacroEngine.kt)
 * **State Machine Runtime**: Build a coroutine-driven state engine using Kotlin SharedFlow and StateFlow primitives. This engine orchestrates transitions between visual detection checks, OCR scans, RL updates, and accessibility gesture dispatches.

### 5. RESOURCE ENGINEERING & SYSTEM ROBUSTNESS
#### 5.1 Thermal & Battery Governance (ThermalGovernor.kt)
 * **Thermal Mitigation**: Monitor system thermals using PowerManager.getThermalHeadroom(30) to predict device temperatures 30 seconds into the future.
 * **Dynamic Throttling**: Automatically scale down performance metrics as thermal or battery limits are reached:
```kotlin
// Monitor Android system thermal states to protect device performance
val currentStatus = powerManager.currentThermalStatus
when (currentStatus) {
    PowerManager.THERMAL_STATUS_LIGHT -> targetFps = 8
    PowerManager.THERMAL_STATUS_MODERATE -> targetFps = 5
    PowerManager.THERMAL_STATUS_SEVERE -> {
        targetFps = 2
        disableReinforcementLearning()
    }
    PowerManager.THERMAL_STATUS_CRITICAL -> haltEngineImmediately()
}

```
 * **Zero-Allocation Hot Paths**: Reuse image buffers and arrays inside the main rendering thread to prevent memory allocations, reducing garbage collection pauses.
#### 5.2 Supervisor Watchdog Daemon (WatchdogService.kt)
 * **Heartbeat Monitoring**: Run a watchdog supervisor inside a separate background process. This process communicates with the primary engine via a lightweight IPC binder interface.
 * **Auto-Recovery**: If the primary automation service is terminated by the OS, the watchdog service will detect the loss of communication, reload the state snapshot from the Proto DataStore, and safely resume the workflow.
